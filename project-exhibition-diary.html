FEATURED PROJECT

展迹 App · 私人观展日记

展迹是一个面向看展人群的私人观展日记 App。用户可以在展览现场快速新建展迹、拍摄展品、补拍作品标签，并在看展结束后把照片、票根、作品信息和感受整理成个人展册。

这个项目的重点不是做一个通用相册，而是把“展厅现场不适合长时间输入”的限制转化成轻量记录流程，让用户先完成低负担采集，再回到安静场景里整理和分享。

PROBLEM DEFINITION

问题定义

看展用户不是不想记录，而是记录动作经常打断观看本身。

为什么选择观展记录

展览现场有强烈的时间、动线和社交限制。用户可能会拍很多照片，但作品名称、作者、年份、票根、地点和当时的感受往往散落在相册、备忘录和社交平台里。

展迹的机会在于把记录拆成两个阶段：现场只做最轻的采集，展后再把内容整理成可回看、可分享的个人展册。

核心断点

  1. 现场拍照很快，但补充作品信息很麻烦。
  2. 展览结束后，照片、票根、感受和作品标签难以归档。
  3. 想分享看展体验，但重新排版长图或九宫格成本高。
  4. 用户希望保留私人记录，不希望所有内容默认公开。
USER SCENARIO

目标用户与使用场景

我把场景收敛在“现场轻记录 + 展后整理”的私人展册链路。

01

看展前

快速建立展览档案，自动带入日期、地点、天气等基础信息。

02

看展中

围绕拍展品、拍标签、记一句和拍票根设计低打扰操作。

03

看展后

把展品、票根、心情和笔记整理成长图、九宫格或封面卡。

MVP STRATEGY

MVP 范围与取舍

优先验证一条完整观展记录链路，不急着做大型社区或展览票务平台。

MVP 包含

  • 新建一条展迹
  • 日期、地点、天气等基础信息录入
  • 展中拍摄展品、标签、票根和一句感受
  • 作品标签 OCR / 待确认机制
  • 展记详情、作品详情和分享图生成

暂不包含

  • 真实票务购买和展览日历
  • 开放社区、关注关系和评论体系
  • 复杂图片编辑器和多端协同
  • 真实 OCR 接口、云备份和账号支付

取舍原因

看展场景的核心不是“功能越多越好”，而是现场操作越轻越好。MVP 先验证用户是否愿意把展览记录集中沉淀到一个专门工具里。

PRD

功能定义

功能围绕“建档、采集、补全、整理、分享”五个动作组织，减少现场输入成本。

展册首页

展示正在展中的记录和最近展记，让用户可以继续未完成的展迹，也能快速回看历史展册。

看展入口

以“新建展迹”和“继续未结束的展”为核心，减少现场寻找入口的负担。

展中采集

把拍展品作为主动作，标签、票根和一句感受作为辅助动作，支持跳过与展后补全。

分享生成

提供长图、九宫格、封面卡三种模式，把私人记录转成适合社交平台分享的视觉内容。

产品页面总览

USER FLOW

核心用户流程

用户先建立展迹，再在现场轻量采集，最后回到展册里整理和分享。

01
进入展册

查看正在展中和最近展记，明确记录沉淀位置。

02
开始看展

展中采集

05

沉淀精选作品、展览信息、感受和作品详情。

06
生成分享图

按长图、九宫格或封面卡导出，完成从私人记录到分享表达的转化。

整理与分享

分享生成页把展览封面、精选展品和观展感受组织成可导出的长图，服务用户的复盘与社交表达。

PRODUCT WORKFLOW

把观展记录拆成产品闭环

这个项目展示的是对真实场景限制的理解：少打扰、可补全、能沉淀、可分享。

01

轻量建档

先建立展览容器，自动带入基础信息，减少用户开局成本。

02

现场采集

把拍展品作为主任务，其他信息允许跳过，避免打断观展。

03

标签识别

作品标签进入待确认状态，把校对留到展后完成。

04

私人展册

按展览维度归档作品、票根、心情和文字记录。

05

分享模板

提供长图、九宫格和封面卡，降低内容二次整理成本。

06

隐私边界

默认私人记录，分享由用户主动触发，避免社区化过早干扰核心体验。
